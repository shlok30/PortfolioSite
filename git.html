Understanding the basics of Git and Github

What is Git?

Git is basically a form of version control used by software engineers for tracking changes made to the source code

Git allows you to have multiple versions of a source code which can act as backups in case any corruption happens

Git also allows developers to collaborate and work together to fix and add new features to the source code simultaneosly without interfering with each others code

Now that we have a basic understanding of what Git does, lets look at the workflow

Repository

A Repository is nothing but a folder that contains all your source code

How is this different from your already created local file folder?

Think of your Repository as the source of all truth. Whenever you write additonal lines of code and hit save on your IDE, you are only making changes to your local file and not your Repository

To make sure that your changes reflect in your Repository as well you need to make a special save request called Commit which we will be covering later in this blog

What is Commit?

It's a special type of save request that we make so that the latest changes in our local folder also reflect in our Repository

Just like the name suggests we only commit changes in our code when we are sure we want it to be a part of our source code

This will help in making sure that only the best and final saved changes actually reflect in your Repository

Creating a Remote Repository

One of the purposes of Git is to allow teams of developers to simultaneosly work on the same code. Having your Repository only be present in your local machine defeats that very purpose

Creeating a Remote Repository which will be present on the internet helps with this issue

The idea is to keep your remote Repository in sync with your local Repository so that all the commits on the local Repository also reflect on the remote Repo

One of the many Remote Repositories is Github

Push Request

Just like the name suggests this request will allow you to push the latest changes(commits) made to your local Repository to your Remote Repository

Branching

Git allows you to create multiple branches from the main source code which allows devs to freely make the necessary changes and commits to the code without directly affecting the main source code branch

For eg. If you were to add a new function which checked if a number is prime or not, you could create a new branch for yourself and then add that function to the code. Then your Manager can review your codes functionality and decide whether it should be added to the main source code or not

This will prevent the main source code from getting corrupted due to any human-error

Pull Request

In order for others in your organisation to see what changes you have made in your branch, you need to do a pull request

This step will decide whether your manager will approve of your changes and allow them to be added to the main soruce code branch

Merge Request

Once your Manager(Owner of Source Code) approves of the changes made in your branch, you can run a merge request which would then add your changes to the original source code

The End
